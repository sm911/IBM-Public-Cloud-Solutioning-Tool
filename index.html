2.2 Sizing Calculations

About the code snippet. The JavaScript shown documents the exact formulas used by the tool (vCPU, vRAM, storage). You don’t run this code. Enter the inputs in the Tool or in the Intake workbook; the tool executes these formulas automatically and shows totals in Solution & Pricing.

vCPU Calculation Formula

Physical cores → subtract buffers → subtract vSAN overhead → add hyper-thread benefit → multiply by vCPU-per-core → multiply by hosts = Total vCPU

cpuCap = server_total_cores cpuCap -= (server_total_cores * cpu_buffer) / 100 if (overhead.incl_buffer) cpuCap -= (serverTotalCores * overhead.cpu_buffer.value) / 100 if (storageType === "VSAN") cpuCap -= (server_total_cores * vsan_overhead) / 100 cpuCap += (server_total_cores * hyperthread_pct) / 100 cpuCap *= vcpu_per_core computeCpuGivenNode = cpuCap * totalNodes total_vcpu = ceil(computeCpuGivenNode)

vRAM Calculation Formula

Host memory → subtract buffers → subtract vSAN RAM overhead → apply oversubscription → multiply by hosts = Total RAM (GiB)

memCap = server_memory if (overhead.incl_buffer) memCap -= (server_memory * overhead.ram_buffer) / 100 if (storageType == "VSAN") memCap -= (server_memory * overhead.vsan_overhead) / 100 memCap *= overhead.ram_oversubscribe computeMemoryCapacity = memCap * totalNodes total_memory = ceil(computeMemoryCapacity)

vSAN Usable Storage Formula

(SSD size × quantity) ÷ data_ratio(RAID, FTT) → apply headroom → apply dedupe → multiply by hosts → convert to TiB

storageCap = vsan.ssd_disk_size.value * vsan.ssd_quantity.value data_ratio = 1 lookup (constraint table -> argu: overhead.raid.value & overhead.ftt.value) -> result if (results.length > 0) { data_ratio = result.ratio } storageCap /= data_ratio if (overhead.incl_buffer.value) { storageCap *= 1 - overhead.hdd_buffer.value / 100 } if (overhead.dedup.value) { storageCap *= overhead.dedup_ratio.value } return storageCap

Supported RAID/FTT Configurations

RAID & FTT quick guide
  • RAID1, FTT1 – 2 copies (mirroring). Highest resiliency, ~50% usable capacity. Use for small clusters or when maximum availability is required.
  • FTT2 – 3 copies. Survives two failures; ~33% usable. Use for critical data or stretched failure domains.
  • FTT3 – 4 copies. Survives three failures; ~25% usable. Rare; typically large clusters with strict SLAs.
  • RAID5, FTT1 – Single‑parity across hosts. Better capacity (~75% usable) with good resiliency. Use when cluster has ≥4 fault domains.
  • RAID6, FTT2 – Dual‑parity. ~66% usable. Recommended for larger clusters where higher protection is required with parity efficiency.
  • RAID 1 FTT 1: ((drive_size_gb × drives_per_node × num_nodes / 2 × (1-headroom) × compression_dedupe_ratio) - 480) / 1000
  • RAID 1 FTT 2: ((drive_size_gb × drives_per_node × num_nodes / 3 × (1-headroom) × compression_dedupe_ratio) - 480) / 1000
  • RAID 1 FTT 3: ((drive_size_gb × drives_per_node × num_nodes / 4 × (1-headroom) × compression_dedupe_ratio) - 480) / 1000
  • RAID 5 FTT 1: ((drive_size_gb × drives_per_node × num_nodes / 1.33 × (1-headroom) × compression_dedupe_ratio) - 480) / 1000
  • RAID 6 FTT 2: ((drive_size_gb × drives_per_node × num_nodes / 1.5 × (1-headroom) × compression_dedupe_ratio) - 480) / 1000

Endurance Storage Formula

Sum of (share size × effective quantity) with headroom applied

let enduranceStorage = 0 for (const fileShare of cluster.storage.endurance) { if (fileShare.quantity.value === 0) { fileShare.quantity.value = 1 } enduranceStorage += fileShare.endurance_size_per_share.value * fileShare.quantity.value } if (cluster.overhead.incl_buffer.value) { enduranceStorage = Math.ceil(enduranceStorage * (1 - cluster.overhead.hdd_buffer.value / 100)) } return enduranceStorage

Note: A quantity of "0" is treated as "1" (minimum one share)

PowerVS (Power Virtual Server) Solutions

3.1 Concepts & Available Patterns

Direct Link → TGW → PER represents Direct Link 2.0 terminating on a Transit Gateway, which then routes to the Power Edge Router (PER) for PowerVS. This is today’s recommended pattern for private connectivity. Existing environments may still use legacy Direct Link 1.0 that connects differently; both are supported.

Direct Link → TGW → PER

Fast private link into PowerVS with on-premises filtering. COS access via VPE or PER. No customer gateway.

DL with Classic Gateway

Single ingress/egress using GRE between TGWs. Gateway on Classic filters traffic. COS via PER or CSE.

Site-to-Site VPN

Easy and cost-effective. Typical throughput up to ~650 Mbps.

Client-to-Site VPN

Remote user access from laptops. Often PowerVS-only access.

Megaport Direct

Customer-procured connection with no IBM Cloud charges. Usually PowerVS-only.

Multi-Region Variants

Add Global TGW and replication method between regions (DL, Classic GW, or Megaport).

Backup Options: The tool includes patterns for IBM i BRMS/ICC, StorSafe VTL, Veeam, and Compass.

Information Gathering Checklist

1 Connectivity: Choose DL, VPN (S2S/C2S), or Megaport with bandwidth requirements
2 Gateways: Need Classic gateway? GRE? Flavor (1 Gbps or 10 Gbps)?
3 VPC Attachment: Attach VPC for VPE or VPN termination?
4 COS Access: Choose path—VPE, PER, or CSE
5 Regions: Single or two? If two: add Global TGW and replication (GRS/app/software)
